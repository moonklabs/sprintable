Chapter 01 · 이야기

출시 전날, 모두가 “완료”라고 말했다

기획서도 있고, 화면도 있고, 코드도 있습니다. 그런데 팀장이 확인을 시작하자 빠진 것이 한꺼번에 보입니다.

PM
신규 가입 화면과 결제 안내까지 오늘 완료할 수 있을까요?
AI
완료했습니다. 화면을 만들고 코드도 수정했으며 PR도 열었습니다.
리드
오류가 날 때 화면은요? 모바일은요? 결제 실패 후 재시도는요?
AI
그 부분은 요구사항에 명시되지 않아 포함하지 않았습니다.
검토하면서 뒤늦게 발견한 것
  • 빈 상태와 오류 상태가 없다
  • 모바일에서 버튼이 가려진다
  • 테스트를 만들었지만 실행하지 않았다
  • 기획서의 “재시도” 조건이 코드에 없다
  • 누가 최종 승인해야 하는지 정해지지 않았다
핵심: 산출물이 생겼다는 사실과, 팀이 결과를 믿고 받아들일 수 있다는 사실은 다릅니다. AI 시대의 “완료”는 생성이 아니라 수용 가능한 근거가 모인 상태입니다.
Chapter 02 · 병목

작업자는 늘었지만 승인자는 늘지 않았다

한 사람이 여러 AI 작업을 동시에 시작할 수 있어도, 중요한 결과를 이해하고 책임지는 사람의 시간은 여전히 하루 8시간입니다.

🔀채팅과 PR을 계속 오간다
🧠매번 요구사항을 다시 떠올린다
⏳완료보다 확인 대기가 길어진다
문제 가설: 코딩 에이전트를 쓰는 개발자와 리드는 “코드를 만드는 속도”가 아니라 여러 에이전트의 결과를 다시 이해하고, 빠진 근거를 찾고, 책임 있게 승인하는 비용 때문에 고통받습니다.

Chapter 07 · 역할별 사례

같은 Ready Check도 직군마다 “아하”가 다르다

아래에서 자신의 역할을 선택해 사람이 확인하기 전에 Advisor가 어떤 일을 덜어주는지 살펴보세요.

기획자 · “내가 쓴 완료 조건이 실제 결과에 들어갔나?”

예전

PR과 채팅을 열어 재시도·권한·예외 조건이 반영됐는지 개발자에게 하나씩 묻습니다.

Ready Check

Linear 완료 조건과 변경 증거를 대조해 “결제 실패 후 재시도 조건의 근거 없음”을 먼저 표시합니다.

효과

코드를 읽지 않아도 어떤 요구가 확인됐고 무엇이 비었는지를 말할 수 있습니다.

디자이너 · “정상 화면 말고 실제 사용 상황도 구현됐나?”

예전

배포 후에야 빈 상태, 오류 상태, 긴 문장, 모바일 잘림, 키보드 접근 문제를 발견합니다.

Ready Check

디자인 체크리스트와 화면 증거를 비교해 빠진 상태와 검증되지 않은 뷰포트를 묶어 보여줍니다.

효과

픽셀 감시가 아니라 사용 경험의 빠진 상태에 집중해 승인할 수 있습니다.

대표·리드 · “내가 봐야 하는 것만 올라왔나?”

예전

여러 프로젝트의 채팅과 PR을 번갈아 열고, 이미 자동으로 고칠 문제까지 직접 지적합니다.

Ready Check

명백한 문제는 먼저 돌려보내고, 고객 영향·범위 변경·배포처럼 사람 판단이 필요한 항목만 올립니다.

효과

리뷰어가 아니라 결정권자로서의 시간을 사용합니다.

개발자 · “다시 모든 맥락을 설명하지 않고 넘길 수 있나?”

예전

AI가 “완료”라고 하면 테스트 실행 여부, 현재 커밋, 요구사항 대응을 다시 캐묻고 복사해 전달합니다.

Ready Check

실행본 SHA, 테스트 결과, 미확인 항목, 재작업 내역을 Work Manifest로 자동 묶습니다.

효과

리뷰 요청 전에 거짓 완료를 스스로 발견하고 수리할 수 있습니다.

Chapter 08 · 제품 형태

새 공장을 짓지 말고, 공통 품질 게이트를 연결한다

Claude와 Codex에는 각자의 작업 방식이 있습니다. Sprintable이 둘을 대체하려 하면 사용자는 옮기지 않습니다. 대신 어디서 일하든 같은 판정 형식을 제공해야 합니다.

🏭권하지 않음 · Full harness

모든 작업을 Sprintable 안에서만 하게 만들기

새 채팅, 새 실행기, 새 프로젝트 관리 방식을 모두 배워야 합니다. 기존 Claude·Codex 생태계와 경쟁하게 됩니다.

🛂권장 · Ready Check runtime

각 도구의 출구에 같은 품질 게이트 연결하기

Claude hook, Codex plugin, MCP에서 같은 check → advise → repair → report 규칙을 호출하고 결과만 Sprintable에 기록합니다.

Plugin은 무엇인가요?

Claude·Codex 같은 도구 안에 Sprintable 명령과 화면을 넣는 작은 확장 기능입니다. 사용자는 익숙한 채팅창을 떠나지 않고 Ready Check를 실행합니다.

MCP는 무엇인가요?

AI가 Linear, Sprintable 같은 외부 도구를 공통 방식으로 읽고 쓰게 해주는 연결 규격입니다. 업무 조회와 결과 보고에 적합하지만, 사람 승인을 대신 증명하지는 않습니다.

Hook은 무엇인가요?

세션 종료, “완료” 보고, PR 생성처럼 특정 사건이 생겼을 때 검사를 자동 실행하는 장치입니다. 매일 쓰게 만드는 핵심은 사용자가 명령을 기억하는 대신 hook이 적절한 순간을 잡는 것입니다.

왜 작은 side library가 필요한가요?

Claude와 Codex의 연결 방식은 달라도 판정 규칙과 출력 형식은 같아야 하기 때문입니다. 단, 작업 실행 전체를 지배하는 harness가 아니라 검증 경계만 담당해야 합니다.

제품 한 줄: sprintable-ready는 AI 실행기가 아니라 어느 실행기에서도 같은 증거 계약과 판정 언어를 쓰게 하는 작은 edge runtime이어야 합니다.

Chapter 10 · 효능 측정

“시간을 줄였다”보다 먼저 관찰할 수 있는 것을 잰다

이제 동료에게 이렇게 설명하세요

Sprintable은 AI를 더 많이 일시키는 도구가 아니라, AI가 만든 일을 사람이 더 적게 의심하고 더 정확히 받아들이게 하는 도구다.

실행은 Claude·Codex에 남겨두고, 의도는 Linear에서 읽고, 실제 변경은 GitHub에서 확인합니다. Sprintable은 이 근거를 한 작업으로 묶어 Advisor가 먼저 누락을 줄이고 사람의 최종 판정을 기록합니다.

Linear는 무엇을 할지 기억하고, Claude·Codex는 만들고, GitHub는 실제 변경을 보관한다. Sprintable은 무엇을 근거로 누가 받아들였는지 기억한다.

5문제로 핵심을 확인해 보세요

정답을 누르면 바로 이유를 확인할 수 있습니다.

1. AI 작업자가 늘 때 가장 먼저 새 병목이 되기 쉬운 것은?

생성은 병렬화되지만 책임 있는 판단은 자동으로 늘지 않기 때문에 수용과 승인 대기열이 병목이 됩니다.

2. Sprintable이 정본으로 가져야 할 것은?

코드는 GitHub, 작업 과정은 Claude·Codex에 둡니다. Sprintable은 의도와 근거, 판정을 연결합니다.

3. Advisor의 올바른 역할은?

Advisor는 선임 검토자입니다. 가치 판단과 책임이 따르는 최종 승인은 사람에게 남습니다.

4. 매일 쓰게 만드는 가장 좋은 방식은?

습관을 새로 요구하기보다 세션 종료·PR 생성 같은 기존 사건에 hook으로 연결해야 반복 사용이 쉽습니다.

5. 파일럿 초기에 가장 믿을 만한 효능 증거는?

첫 단계에는 관찰 가능한 rescue 사례와 재발률을 측정해야 합니다. 타임스탬프만으로 절감 시간을 단정하면 안 됩니다.
