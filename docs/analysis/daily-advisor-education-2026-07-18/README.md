# Sprintable Daily Advisor 교육자료

Claude·Codex·Linear·GitHub을 함께 쓰는 팀에서 왜 `완료 확인과 승인`이 새 병목이 되는지, Sprintable이 어떤 역할을 맡아야 하는지를 비개발자도 설명할 수 있도록 만든 교육 패키지다.

## 대상

- 프로덕트 매니저·서비스 기획자
- 프로덕트·브랜드·UX 디자이너
- 엔지니어 출신 대표·CTO·팀 리드
- AI 코딩 에이전트 도입을 검토하는 비개발 직군

## 학습 목표

교육 후 참가자는 다음을 자신의 말로 설명할 수 있어야 한다.

1. AI 시대의 병목이 왜 `만들기`에서 `믿고 받아들이기`로 이동하는가.
2. Linear, Claude/Codex, GitHub, Sprintable이 각각 무엇의 정본이어야 하는가.
3. `컨텍스트를 잘 보내는 것`과 `판정 기록을 잘 쌓는 것`의 차이는 무엇인가.
4. Advisor가 사람을 대체하지 않고 사람에게 오기 전 실수를 줄이는 구조는 무엇인가.
5. 왜 Sprintable은 별도 full harness가 아니라 작은 공통 Ready Check runtime을 가져야 하는가.

## 파일

- `index.html` — 10~15분 자율 학습용 본문. 역할별 사례, 시각 흐름, 용어 설명, 확인 퀴즈 포함.
- `facilitator-guide.md` — 20분 팀 교육 진행안과 토론 질문.
- `workshop-sheet.md` — 실제 업무 하나를 Ready Check 흐름으로 바꾸는 15분 실습지.
- `sources.md` — 저장소 근거, 외부 공식 자료, 정량 수치 사용 주의사항.
- `cto-linear-gate-advisor-implementation-proposal.md` — 개발 CTO를 첫 고객으로 두고 Linear-first 온보딩, 최소 MCP/API 변경, 승인 무결성, 6주 구현 순서를 정리한 개발 착수 제안.

## 추천 사용법

### 자율 학습

`index.html`을 브라우저에서 열고 역할별 사례와 퀴즈까지 진행한다.

### 팀 세션

1. `index.html` 1~6장을 함께 읽는다. — 10분
2. 역할별 사례에서 참가자 직군에 가까운 탭을 연다. — 3분
3. `workshop-sheet.md`로 실제 task 한 건을 변환한다. — 15분
4. facilitator가 `어디에서 사람 승인이 꼭 필요한가`를 정리한다. — 5분

## 한 문장

> Linear는 무엇을 할지 기억하고, Claude·Codex는 만들고, GitHub는 실제 변경을 보관한다. Sprintable은 무엇을 근거로 누가 받아들였는지를 기억한다.
