Sprintable · 설계 플랜 리포트 · 2026-07-19

Advisor는 누가, 언제 부르나 — 호출 모델 설계

백엔드(advisor_context · report_done · gate.resolved)는 이미 완성돼 있습니다. 비어 있는 것은 "누가 언제 호출하는지"를 정의하는 레이어 — 이 리포트는 그 레이어를 Skill·Command·Hook 세 부품으로 어떻게 만들지의 계획입니다.

핵심 답 한 문장

개발자(사람)는 아무 툴도 직접 호출하지 않는다 — 에이전트가 호출하고, 스킬이 에이전트에게 "언제 호출할지"를 가르친다.

Skill = 자동(세션의 기본 동작) · Command = 수동 진입점(시작·강제·확인) · Hook = 나중에(불가역 경계 강제, 가설 확증 후). 사람이 하는 일은 단 하나 — 게이트에서 증거를 보고 싸인하는 것.

15단계 루프 — 무엇이 자동이고 무엇이 수동인가

에이전트가 티켓 하나를 처리하는 전체 흐름. 색이 곧 답입니다.

자동 — 에이전트·스킬 수동 — 사람 개입 없음
①

티켓 수신

진입은 수동이후 자동

사람이 /sprintable:work SPR-31을 입력하거나, 에이전트의 poll_events 루프에 배정 이벤트가 도착하면 시작됩니다.

agent> SPR-31 시작합니다. AC 3개 확인.
②

Kickoff Advisor

자동 (Skill)

스킬 규범에 따라 에이전트가 advisor_context(kickoff)를 호출 → 서버가 내려준 프롬프트를 자기 로컬 서브에이전트로 실행합니다. 서버는 모델을 돌리지 않습니다. 과거 이 프로젝트에서 사람이 reject한 이력이 작업 계획에 반영됩니다.

advisor> 과거 reject 이력 2건 참고: AC마다 근거 필수.
③

작업

Sprintable 불개입

코딩은 지금처럼 harness(Claude Code·Codex) 안에서. 쓰던 도구를 하나도 바꾸지 않습니다.

④

Preflight → 완료 보고

자동 (Skill)/sprintable:done 으로 강제 가능

에이전트가 "완료"라고 판단하면 스킬이 지시합니다: advisor_context(preflight) → 서브에이전트가 self_review(verdict·findings·keep)를 생성 → report_done에 첨부. merge 단계면 게이트가 판단해 200(자동 통과) / 202(사람 대기) / 409(차단)으로 답합니다. self_review는 주장이지 승인이 아닙니다 — 게이트에 증거로 봉인될 뿐.

agent> self_review: likely_pass · findings 1건 → 202, 사람 게이트 대기. 다른 티켓 진행합니다.
⑤

판정 → 재작업

싸인은 사람수신·재작업은 자동

사람이 게이트 화면에서 증거를 보고 approve/reject — 루프 전체에서 사람이 하는 유일한 일입니다. 판정은 gate.resolved 이벤트로 에이전트에게 돌아가고, rejected면 note·keep을 준수하며 자동으로 재작업 후 ④로 복귀합니다.

agent> rejected · next_stage=revise — keep 목록 보존하고 재작업 시작.

2세 부품의 역할 분담

같은 루프를 세 아티팩트가 나눠 구동합니다 — 겹치지 않고, 하나가 빠져도 나머지로 돌아갑니다.

Skill

자동

세션의 기본 동작 — 루프 규범

  • 티켓 시작 시 kickoff advisor 실행
  • 완료 판단 시 preflight → report_done
  • 202는 대기가 아니라 다른 티켓 진행 신호
  • rejected 수신 시 keep 보존 재작업 규칙
  • 금지 규칙: self_review ≠ 승인, 주입 방어

Command

수동

사람의 명시적 진입점 3개

  • /sprintable:work — 티켓 시작
  • /sprintable:done — 완료 선언 강제 실행
  • /sprintable:status — 게이트·이벤트 확인
  • 스킬이 안 도는 상황의 탈출구 + 데모용

Hook

P2 연기

불가역 경계에서만 강제

  • git push / merge 직전 preflight 없으면 차단
  • H-도장 가설 확증(7/31) 후에만 도입
  • 안티 세리머니 — 확증 전에 강제부터 넣지 않는다

3도그푸드 시나리오 — 실제 티켓 1건의 대화

Phase 1의 수용 테스트이자 H-영수증 가설의 첫 계측(게이트 open→resolve 시간).

사람

/sprintable:work SPR-31 — 루프에서 사람이 입력하는 첫 번째이자 (판정 전까지) 마지막 명령

에이전트

advisor_context(kickoff) 호출 → 서브에이전트 실행 → "reject 이력 2건 참고" 반영 후 작업 시작

에이전트

작업 완료 판단 → advisor_context(preflight) → self_review 생성 → report_done(self_review)

서버

202 — 사람 게이트 생성, self_review를 증거로 봉인(advisor_origin). 에이전트는 다른 티켓으로.

사람

게이트 화면에서 증거 확인 → reject + note 입력 (예: "권한은 project_auth 헬퍼로")

에이전트

poll_events에 gate.resolved(rejected, next_stage=revise) 도착 → 자동 재작업 → 두 번째 report_done

사람

approve — 싸인. 티켓 done, 판정 근거는 원장에 영구 기록

4아하 모먼트 — 언제 "이거다"를 느끼나

세 개가 있고, 오는 시점이 다릅니다. 온보딩의 진짜 목적지는 MCP 등록이 아니라 아하 1입니다.

아하 1 · 첫 티켓, 첫 5분

코드를 안 열었는데 판정이 끝났다

첫 202 게이트 화면 — self_review와 증거가 나란히 붙어 있어 3초 안에 approve. "다 했어요"를 의심하며 전부 다시 열던 사람이 처음으로 안 열어본 순간. H-영수증(판정 중앙값 <5분)이 정확히 이 순간을 계측합니다.

가장 강한 순간 · 데모의 하이라이트
아하 2 · 첫 reject, 같은 날

reject했더니 복붙 없이 알아서 고쳐왔다

note 한 줄 적고 reject → gate.resolved가 날아가고, 에이전트가 keep은 보존한 채 재작업한 두 번째 패킷이 돌아온다. 리뷰 코멘트를 터미널에 복붙하던 루프가 사라지는, 루프가 닫혔다는 감각. PoD 설계에 사전 등록된 "가장 보여줘야 할 장면".

아하 3 · 티켓 3~5개 후 (리텐션 훅)

내 싸인이 데이터가 됐다

kickoff advisor가 "과거 reject 이력 2건 참고"라고 말하는 순간 — 내가 찍은 판정이 다음 작업의 스펙으로 되먹임되는 걸 목격. 첫날엔 못 느끼고(이력이 없으니), 쓸수록 좋아진다는 신호라 리텐션을 만드는 아하.

아하가 아닌 것: kickoff advisor의 첫 출력 자체(똑똑한 컨텍스트 주입으로 보일 뿐), 42% 숫자(광고의 훅이지 제품의 아하가 아님). — 설계 함의: ① 도그푸드·데모 시나리오에 reject를 일부러 한 번 넣는다(happy path로 가면 아하 2를 못 보여줌 — §3 시나리오에 반영됨). ② 온보딩은 "첫 202 게이트"까지의 시간을 최단으로.

5실행 계획 — 4개 Phase

Phase 00.5일

백엔드 착지. 워킹 트리의 Advisor P0 + gate.resolved 코드를 develop 기준 feature/advisor-p0 브랜치로 분리 커밋 → 테스트 10개 파일 통과 확인 → PR.

Phase 11일 · 이번 주

Skill + Command 작성, 도그푸딩. 이 repo .claude/에 SKILL.md 1개 + 커맨드 3개 작성 → 자사 org에 플래그 활성화(provenance 사전검사 포함) → 실제 티켓 1건 풀루프(위 시나리오)로 수용 테스트.

Phase 21~2일

배포 패키징. packages/claude-plugin/ 신설, CLI connect가 MCP 설정과 함께 스킬을 자동 설치 — "MCP 한 줄 + 스킬 자동 설치"로 TTHW<5분 유지. quickstart 문서에 루프 다이어그램 추가.

Phase 3가설 게이트

후행 — 결과 나와야 착수. PreToolUse Hook(불가역 경계 강제)은 H-도장 확증(7/31) 후. Codex 변형(AGENTS.md + codex exec 레시피), AC별 self_assessment 구조화는 별도 세션.

6검증 — 무엇이 되면 "된 것"인가

  • Phase 0: advisor 테스트 10개 파일 + MCP contract 테스트 전체 통과
  • 도그푸드 E2E: 완료 선언 시 스킬이 사람 개입 없이 preflight → report_done을 실행하고 202 + gate_id가 확인된다
  • 웹 게이트 화면에 advisor_origin 증거가 표시되고, reject 후 gate.resolved(revise)가 에이전트에 도착해 재작업이 자동 시작된다
  • 계측: 게이트 open→resolve 시간 기록 — H-영수증 목표는 판정 중앙값 5분 이내
